# Lan Code 多模态能力路由

## 目标

Lan Code 不把“文字模型、图片模型、语音模型”做成互相隔离的配置页，而是把每个模型视为
一组能力的集合。客户端和 Core 只请求能力，路由层负责选择实际模型。

## 能力模型

当前已建立以下能力：

- `toolCalling`：工具调用
- `imageInput`：图片理解
- `imageOutput`：图片生成
- `audioInput`：语音识别
- `audioOutput`：语音输出

后续可以继续加入视频、音乐、Embedding、重排序和 OCR，而不破坏现有配置结构。

## 路由规则

1. 首先检查当前主模型是否声明或检测到目标能力。
2. 主模型支持时，默认直接复用主模型配置。
3. 主模型不支持时，检查用户是否启用了该能力的专用路由。
4. 专用路由的 API 地址或 API Key 留空时，可以继承主 Provider。
5. 没有可用路由时，客户端应明确提示缺少哪项能力，不能静默失败。

## 自动检测

第一版能力检测由两部分组成：

- 实际请求：文本响应和工具调用通过真实 API 请求验证。
- 模型信息推断：根据常见模型命名识别图片和语音能力。

模型命名只能作为保守推断。后续应优先使用 Provider 模型列表中的能力元数据，并增加真实
图片输入、图片生成、语音输入和语音输出探测。

## 安全要求

- 专用能力 API Key 与主 API Key 一样属于敏感信息。
- 生成的媒体文件必须保存到工作区或用户明确选择的数据目录。
- 任何把文件上传给模型的行为都必须在 UI 中清楚说明。
- 自动能力探测不能产生高额调用费用或不可逆外部副作用。
