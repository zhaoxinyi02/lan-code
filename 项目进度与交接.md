## 一、项目基本信息

Lan Code 是一个本地优先、客户端无关的 AI 编程 Agent。核心使用 Rust，
桌面端使用 Tauri + React。未来计划支持 CLI、桌面端、VS Code、JetBrains 和 Web。

核心原则：

1. Agent 真相属于 core，不属于桌面 UI。
2. 模型只能提出工具调用，权限与执行由 core 控制。
3. 会话、工具、审批和事件必须可持久化、可恢复。
4. 默认保护用户文件；高风险命令必须明确授权。
5. 所有面向用户和开发交接的文档优先使用中文。

主要目录：

| 路径 | 作用 |
| --- | --- |
| `crates/lan-protocol` | 跨客户端协议类型 |
| `crates/lan-core` | Agent loop、模型适配、工具、权限、存储 |
| `crates/lan-daemon` | JSONL daemon，为未来多客户端提供边界 |
| `crates/lan-cli` | CLI 客户端 |
| `apps/desktop` | Tauri + React 桌面端 |
| `docs/office-mode.zh-CN.md` | Office Mode 中文说明 |
| `research/repos` | 18 个参考项目的浅克隆，不作为依赖 |
| `docs/research-analysis.zh-CN.md` | 参考项目研究结论 |

## 二点六、2026-06-16 多模型与 Agent 强化记录

参考 Codex、Cline、Roo Code、OpenCode、Kun 后，本轮落地重点是“用户可接任意模型”带来的兼容问题：

- Core 新增模型上下文窗口配置，桌面端会把当前模型配置的 `contextWindow` 传入 core。
- Core 在发送给模型前做保守上下文压缩副本：保留系统提示和最近消息，把旧历史折叠成系统摘要提示，避免切换小上下文模型后直接爆上下文。
- Core 不破坏本地完整会话历史；压缩只作用于本次模型请求。
- Core 系统提示增强：要求先读仓库、不要编造文件内容、倾向小步修改、上下文不足时先压缩。
- OpenAI-compatible provider 开始尊重模型配置中的 `maxOutputTokens`，写入请求的 `max_tokens`。
- Provider 错误提示增强：识别上下文超限、鉴权失败、模型不存在、限流等常见错误，并给中文修复方向。
- 桌面端发送消息前检测上下文使用量：超过阈值会提示用户，并注入“先压缩旧上下文再处理”的结构化指令。
- 桌面端切换模型时检测目标模型上下文窗口：若当前对话已超过目标窗口，提示用户切换后会先压缩历史。
- 模型切换菜单显示每个模型的上下文窗口，例如 `128K 上下文`、`1M 上下文`。
- 用户消息中的 GitHub 仓库 URL 会被识别为仓库资源，并在本轮提示里明确告知 Agent。
- Agent 回复中的 GitHub 仓库链接会渲染为 GitHub 图标 + `用户名/仓库名`。
- Agent 回复中用反引号包裹的工作区文件路径会显示为蓝色文件引用，并带真实文件类型图标。
- 全局字体栈增强，优先使用苹方、MiSans、HarmonyOS Sans SC 等更现代的中文字体。
- 快速任务新增“压缩上下文”和“研究仓库”两个模板，方便做长任务交接和参考仓库研究。

## 四、本轮用户反馈与决策

用户反馈：

1. Windows 安装器触发 SmartScreen。
2. 长任务生成了一部分文件，但最后因超过 12 轮被判失败。
3. 功能太少，需要多模型、模型可用性测试、项目管理、丰富设置、删除会话。
4. 默认数据目录应为用户主目录下的 `.lancode`，并允许用户自行选择。
5. 用户接受 API Key 明文保存在数据目录配置文件中。
6. 希望加入 Token 计费和更多 Kun 类能力。

当前产品决策：

- 默认数据目录改为 `~/.lancode`。
- 支持用户选择自定义数据目录。
- API Key 可明文写入用户选择的数据目录；UI 必须明确提示风险。
- Provider 先统一走 OpenAI-compatible 协议，并提供常用预设；之后再增加
  Anthropic/Gemini 原生能力适配。
- SmartScreen 的根本解决方案是购买并配置 Windows 代码签名证书。未签名版本只能
  提供 SHA-256、发布说明和风险提示，不能消除 SmartScreen。
